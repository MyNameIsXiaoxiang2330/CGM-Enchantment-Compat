# 与官方 CGM 的附魔差异分析

> 本文档详细解释本模组与官方高版本 CGM (1.0.1+) 附魔系统的差异、原因，以及未实现/有 Bug 功能的说明。
> 最后更新: 2026-07-09，v0.0.8

---

## 为什么会产生这些差异？？？

本模组的目标平台是 **Minecraft 1.12.2 + CGM v0.15.3**。
高版本 CGM (1.0.1+) 经历了大规模重写：
- 从 `Forge` 迁移到 `Framework`（自研加载器）
- 引入数据驱动系统（JSON 配置 → 数据包）
- 重写了实体碰撞、弹药系统、附魔系统
- 引入了暴击、配件系统等新机制

**CGM v0.15.3 没有这些机制**，所以我们只能在现有架构下用不同的方式实现"类似"的效果。

---

## 附魔全对比

### 1. 穿甲弹 (Puncturing)

| 项目 | 官方 CGM 1.0.1+ | 本模组 |
|------|-----------------|--------|
| 效果 | 暴击率 +5%/级 | **忽视护甲 25%/级** |
| 最高等级 | IV | IV |

**产生差异的原因？**

CGM v0.15.3 **没有暴击系统**。暴击机制是 CGM 1.0+ 版本才引入的，它依赖于 Framework 的事件系统和数据驱动中的 `damage` 计算链。在 1.12.2 的 CGM 中，子弹的伤害计算是硬编码在 `EntityBullet` 中的，无法在不修改字节码的情况下插入暴击判定。

**我们的替代方案**：利用原版 Minecraft 的护甲机制，直接忽视目标护甲值。这不需要修改 CGM 的伤害计算链，只需在 `LivingHurtEvent` 中调整最终伤害即可实现。

### 2. 间接伤害 (Collateral)

| 项目 | 官方 CGM 1.0.1+ | 本模组 |
|------|-----------------|--------|
| 效果 | 子弹穿透实体，伤害不减 | 子弹穿透实体，伤害不减 |
| 最高等级 | **I** | **III** |
| 粒子效果 | 无 | **白色烟花粒子** |
| 是否一致 | — | 还原，但原理和作用有所改动 |

**产生差异的原因？？**

官方 CGM 将间接伤害锁定为 I 级（只有穿透，不缩放）。我们增加了 II~III 级（每级伤害加成 +20%），这是因为：
- 我们的穿透系统不修改 CGM 的实体碰撞逻辑，而是通过 `EntityJoinWorldEvent` 拦截子弹后再用**独立射线**进行穿透检测
- 独立射线允许我们自由控制伤害缩放
- 粒子效果（`FIREWORKS_SPARK`）是额外的视觉反馈

**MOD所有者注**
我的意图是让“有这个附魔的枪械直接删掉子弹，然后直接计算在这条线上所贯穿的实体，并在命中实体方块/未加载区域做结束，同时用粒子效果作为可见的视觉引导”
这个功能本身实现的很快，但是为了渲染粒子效果我不得不在这上面花费不少Token。
为了防止可能会发生的严重卡顿事故，所有的子弹（或虚拟的子弹）我都会要求在不能加载的地方干脆的删除。

**实现原理**：
```
子弹生成 → BulletHandler 捕获 → 记录射击者+方向+伤害
         → 丢弃原始子弹（CollateralHandler 接管）
         → 每 tick 推进射线 1.5 格
         → AABB 碰撞检测 → 命中实体 → 造成伤害 → 继续穿透
```

### 3. 快速扳机 (Trigger Finger)

| 项目 | 官方 CGM 1.0.1+ | 本模组 |
|------|-----------------|--------|
| 状态 | ✅ 正常 | ❌ **未实现** |
| 原因 |  CGM v0.15.3 冷却系统不可拦截 |

**为什么做不了？**

CGM v0.15.3 的射速由枪械 JSON 中的 `general.rate` 字段控制，该值在类加载时从 JSON 解析并缓存到 `Gun` 对象中。开火判定流程如下：

```
1. 玩家扣动扳机
2. CGM 检查 CommonEvents.getCooldownTracker(UUID) 的剩余冷却
3. 如果冷却 > 0 → 不开火
4. 如果冷却 ≤ 0 → 开火，重置冷却 = rate
```

**问题**：冷却值在 CGM 内部方法中被直接读取和比较，我们无法通过：
- **反射**：无法在运行时修改私有字段的访问路径
- **CoreMod ASM**：可以注入，但冷却检查在多个方法中硬编码，需要修改 4+ 个方法，冲突风险极高
- **事件监听**：Forge 没有"冷却即将检查"的事件

**替代方案**（未来考虑）：
- 直接 ASM 修改 `rate` 的读取路径（高风险）
- 在 CGM 的冷却检查点注入自定义 Hook

目前从附魔台移除，保留注册以防存档损坏。

**MOD所有者注**
但这个附魔仍然在创造模式物品栏里面，我可能会在未来干脆的删掉。

### 4.勤俭节约（原弹药回收）

| 最高等级 | III | III |

## 技术实现原理

### CoreMod 注入点

本模组的 CoreMod 在类加载时修改 CGM 的两个类：

```
GunTransformer (IFMLLoadingPlugin)
  └─ GunClassTransformer (IClassTransformer)
       ├─ ItemGun → 注入 getItemEnchantability() → 返回 30
       └─ ReloadTracker.isWeaponFull() → 替换弹容量读取路径
```

### 事件驱动架构

所有附魔效果通过 Forge 事件总线实现，不修改任何 CGM 字节码（除上述两处注入外）：

| 事件 | 处理器 | 处理附魔 |
|------|--------|---------|
| `LivingDeathEvent` | SinBulletHandler | FELLBULLET 系列 |
| `LivingHurtEvent` | DamageHandler | 加速器、穿甲弹、纵火者、弧光引导、高爆弹 |
| `LivingHurtEvent` | HighExplosiveHandler | 高爆弹（额外爆炸） |
| `EntityJoinWorldEvent` | BulletHandler | 加速器（弹速）、纵火者（追踪）、高爆弹（霰弹射程）|
| `EntityJoinWorldEvent` | CollateralHandler | 间接伤害（穿透） |
| `EntityJoinWorldEvent` | HighExplosiveHandler | 高爆弹（火箭追踪） |
| `PlayerTickEvent` | GunStateHandler | 超容量、弹药回收、熟练手 |
| `PlayerTickEvent` | PlayerHandler | 轻装上阵（移速+散布） |
| `ExplosionEvent.Detonate` | ArcLightHandler | 弧光引导（爆炸AOE） |
| `WorldTickEvent` | 多个 Handler | 子弹追踪、射线推进、蓄力动画 |
